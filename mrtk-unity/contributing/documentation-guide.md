## <a name="style"></a>Стиль

### <a name="writing-style"></a>Стиль письма

Общее правило бегунка: попробуйте использовать **Sound Professional**. Это обычно означает, что следует избегать «диалогового тона». Также старайтесь не использовать гиперболический и сенсатионалисм.

1. Не пытайтесь (слишком) веселое.
2. Никогда не писать "I"
3. Избегайте "мы". Обычно это можно легко заменять, используя вместо этого "МРТК". Пример: "Мы поддерживаем эту функцию"-> "МРТК поддерживает эту функцию" или "поддерживаются следующие функции...".
4. Аналогичным образом старайтесь не использовать. Пример: "благодаря этому простому изменению шейдер станет настраиваемым!" -> "шейдеры могут быть настроены с минимальными усилиями".
5. Не используйте фразы "сыром".
6. Старайтесь не подавать никакого звука, поэтому не нужно ничего продавать.
7. Аналогичным образом Избегайте чрезмерного существенного. Восклицательные знаки редко требуются.

### <a name="capitalization"></a>Регистр букв

- Используйте **предложение для заголовков**. Пример. делать первые буквы и имена прописными, но не предпринимайте никаких других действий.
- Используйте обычную английскую версию для всех остальных. Это означает, что не следует заменять **прописными буквы произвольными словами**, даже если они содержат специальное значение в этом контексте. Предпочитать *текст курсивом* для выделения определенных слов [см. ниже](#emphasis-and-highlighting).
- Если ссылка внедрена в предложение (предпочтительный метод), стандартное название главы всегда использует прописные буквы, тем самым нарушая правило отсутствия произвольного капитализации внутри текста. Поэтому для исправления регистра используйте пользовательское имя ссылки. Например, ниже приведена ссылка на документацию по [элементу управления Bounds](../features/ux-building-blocks/bounds-control.md) .
- Заменяйте прописные имена, например *Unity*.
- При написании *редактора Unity* не используйте прописные буквы "редактор".

### <a name="emphasis-and-highlighting"></a>Выделение и выделение

Есть два способа подчеркнуть или выделить слова, сделав их полужирным шрифтом или сделав их курсивом. Выделение полужирным шрифтом заключается в том, что **полужирный текст фиксируется** и, следовательно, может быть легко замечен при скимминг фрагмента текста или даже при простом прокрутке страницы. Полужирный шрифт прекрасно подходит для выделения фраз, которые следует запомнить. Однако **полужирный текст редко используется**, так как обычно отвлекает внимание.

Часто требуется, чтобы группа была логически объединена или выделяющая определенный термин, поскольку имеет специальное значение. Такие вещи не нужно выделять из общего текста. Используйте курсивный текст в качестве *упрощенного метода* , чтобы выделить что-нибудь.

Аналогично, когда в тексте упоминается имя файла, путь или запись меню, желательно сделать его курсивом, чтобы логически сгруппировать его, не отвлекаясь.

Как правило, старайтесь **избегать ненужных выделений текста**. Специальные термины можно выделить один раз, чтобы сделать его осведомленным, не повторяйте такое выделение во всем тексте, если оно больше не работает и только отвлекается от него.

### <a name="links"></a>Ссылки

Вставляйте как можно больше полезных ссылок на другие страницы, но только один раз. Предположим, что читатель щелкает каждую ссылку на странице и считает, насколько неудобно, если одна и та же страница открывается 20 раз.

Предпочитать ссылки, внедренные в предложение:

- Ошибка: рекомендации полезны. Дополнительные сведения см. в [этой главе](../contributing/documentation-guide.md) .
- ХОРОШЕЕ: [рекомендации](documentation-guide.md) полезны.

Избегайте внешних ссылок, они могут устареть или содержать содержимое с авторскими правами.

При добавлении ссылки определите, следует ли также добавить ее в список в разделе " [см. также](#see-also) ". Аналогичным образом проверьте, следует ли добавить ссылку на новую страницу в связанную страницу.

### <a name="images--screenshots"></a>Изображения и снимки экрана

**Используйте снимки экрана с осторожностью.** Обслуживание образов в документации является большим объемом работы, но небольшие изменения пользовательского интерфейса могут привести к небольшому количеству снимков экрана. Следующие правила снижают затраты на обслуживание:

1. Не используйте снимки экрана для тех вещей, которые можно описать в тексте. В частности, **никогда не следует отображать сетку свойств** с целью отображения имен и значений свойств.
2. Не включайте в снимок экрана элементы, которые не нужны для отображения. Например, когда отображается результат отрисовки, сделайте снимок экрана окна просмотра, но исключите для него пользовательский интерфейс. Отображение пользовательского интерфейса. Попробуйте переместить окна таким образом, чтобы в изображении был только этот важный компонент.
3. При включении пользовательского интерфейса снимка экрана отображаются только важные части. Например, если речь идет о кнопках на панели инструментов, создайте небольшое изображение, отображающее важные кнопки на панели инструментов, но исключите все вокруг нее.
4. Используйте только те образы, которые легко воспроизвести. Это означает, что не следует рисовать маркеры или выделения на снимках экрана. Во – первых, не существует правил, определяющих, как они должны выглядеть. Во-вторых, воссоздание такого снимка экрана является дополнительными усилиями. Вместо этого Опишите важные части в тексте. Существуют исключения из этого правила, но они редки.
5. Очевидно, что создавать анимированный GIF-файл гораздо сложнее. Предполагалось, что он должен быть ответственным за его повторное создание до окончания времени или ожидать, что пользователи не хотят тратить это время.
6. Постарайтесь, чтобы количество образов в статье было низким. Часто хорошим методом является создание одного общего снимка экрана некоторого средства, показывающего все, а затем описание остальной части текста. Это позволяет легко заменить снимок экрана при необходимости.

Некоторые другие аспекты:

- В пользовательском интерфейсе редактора снимков экрана должен использоваться светло-серый редактор тем, что не все пользователи имеют доступ к темной теме, и нам хотелось бы, чтобы они были как можно более постоянными.
- Ширина изображения по умолчанию — 500 пикселей, так как она хорошо отображается на большинстве мониторов. Старайтесь не отклоняться от него слишком много. ширина 800 пикселей должна быть максимально допустимой.
- Используйте Пнгс для снимков экрана пользовательского интерфейса.
- Используйте Пнгс или Жпгс для снимков экрана с трехмерным окном просмотра. Предпочтение степени качества сжатия.

- [Свойства, методы, блоки сводки событий](#property-method-event-summary-blocks)

- [Сериализованные поля](#serialized-fields)

В дополнение к приведенному выше коду следует прокомментировать код, чтобы обеспечить возможность обслуживания, исправления ошибок и простоту настройки.

### <a name="class-struct-enum-summary-blocks"></a>Класс, структура, перечисление блоков сводки

Если в МРТК добавляется класс, структура или перечисление, необходимо описать его назначение. Это следует сделать в виде блока сводки над классом.

```c#
/// <summary>
/// AudioOccluder implements IAudioInfluencer to provide an occlusion effect.
/// </summary>
```

Если имеются какие-либо зависимости уровня класса, их следует задокументировать в блоке «примечания» непосредственно под сводкой.

```c#
/// <remarks>
/// Ensure that all sound emitting objects have an attached AudioInfluencerController.
/// Failing to do so will result in the desired effect not being applied to the sound.
/// </remarks>
```

Запросы на вытягивание, отправленные без сводок для классов, структур или перечислений не будут утверждены.

### <a name="property-method-event-summary-blocks"></a>Свойства, методы, блоки сводки событий

Свойства, методы и события (ПМЕС), а также поля задокументированы с помощью сводных блоков, независимо от видимости кода (общедоступный, частный, защищенный и внутренний). Средство создания документации отвечает за фильтрацию и публикацию только общедоступных и защищенных компонентов.

Примечание. блок сводки **не** требуется для методов Unity (например: "спящий", "запустить", "Обновить").

Для утверждения запроса на вытягивание **требуется** документация по PME.

Как часть сводного блока PME, требуется значение и назначение параметров и возвращаемых данных.

```c#
/// <summary>
/// Sets the cached native cutoff frequency of the attached low pass filter.
/// </summary>
/// <param name="frequency">The new low pass filter cutoff frequency.</param>
/// <returns>The new cutoff frequency value.</returns>
```

### <a name="feature-introduction-version-and-dependencies"></a>Версия и зависимости введения функции

В составе сводной документации по API содержатся сведения о версии МРТК, в которой была представлена эта функция, а также о любых зависимостях, которые следует задокументировать в блоке примечаний.

Зависимости должны включать расширения и (или) зависимости платформы.

```c#
/// <remarks>
/// Introduced in MRTK version: 2018.06.0
/// Minimum Unity version: 2018.0.0f1
/// Minimum Operating System: Windows 10.0.11111.0
/// Requires installation of: ImaginarySDK v2.1
/// </remarks>
```

### <a name="serialized-fields"></a>Сериализованные поля

Рекомендуется использовать атрибут ToolTip Unity, чтобы предоставить документацию среды выполнения для полей сценария в инспекторе.

Так что параметры конфигурации включены в документацию по API, скрипты должны включать *по крайней мере* содержимое подсказки в блок сводки.

```c#
/// <summary>
/// The quality level of the simulated audio source (ex: AM radio).
/// </summary>
[Tooltip("The quality level of the simulated audio source.")]
```

### <a name="enumeration-values"></a>Значения перечисления

При определении и перечислении код должен также документировать значение перечислимых значений с помощью блока Summary. Блоки примечаний при необходимости можно использовать для предоставления дополнительных сведений для улучшения понимания.

```c#
/// <summary>
/// Full range of human hearing.
/// </summary>
/// <remarks>
/// The frequency range used is a bit wider than that of human
/// hearing. It closely resembles the range used for audio CDs.
/// </remarks>
```

## <a name="how-to-documentation"></a>Документация с инструкциями

Многим пользователям набора средств Mixed Reality может не потребоваться использовать документацию по API. Эти пользователи будут использовать готовые, многократно используемые Prefabs и сценарии для создания своих возможностей.

Каждая область функций будет содержать один или несколько файлов Markdown (. md), которые описывают на достаточно высоком уровне, что предоставляется. В зависимости от размера и (или) сложности определенной функциональной области может потребоваться наличие дополнительных файлов, вплоть до одной для каждой предоставляемой функции.

При добавлении компонента (или изменении его использования) необходимо предоставить обзорную документацию.

Как часть этой документации, вы можете использовать разделы с инструкциями, включая иллюстрации, чтобы помочь клиентам в создании новой функции или концепции в разделе Приступая к работе.

## <a name="design-documentation"></a>Документация по проектированию

Смешанная реальность дает возможность создавать совершенно новые мировые возможности. В результате может потребоваться создание пользовательских ресурсов для использования с МРТК. Чтобы сделать это как можно более свободным для клиентов, компоненты должны предоставить документацию по проектированию, описывающую форматирование или другие требования для художественных ресурсов.

Ниже приведены некоторые примеры, в которых документация по разработке может быть полезной:

- Модели курсоров
- Визуализации пространственных сопоставлений
- Файлы звуковых эффектов

Этот тип документации **настоятельно** рекомендуется и **может** быть запрошен в ходе проверки запроса на включение внесенных изменений.

Это может отличаться от рекомендаций по проектированию на [сайте MS Developer](/windows/mixed-reality/design) .

## <a name="performance-notes"></a>Заметки о производительности

Некоторые важные функции имеют стоимость производительности. Часто этот код будет очень в зависимости от того, как они настроены.

Пример.

```md
When using the spatial mapping component, the performance impact will increase with the level of detail requested.  
It is recommended to use the least detail possible for the desired experience.
```

Заметки о производительности рекомендуются для компонентов ЦП и (или) ресурсов с большим количеством GPU и **могут** быть запрошены в ходе проверки запроса на включение внесенных изменений. Все применимые заметки о производительности должны быть включены в API **и** обзорную документацию.

## <a name="breaking-changes"></a>Критические изменения

Документация по критическим изменениям состоит из [файла](../contributing/breaking-changes.md) верхнего уровня, который ссылается на отдельные breaking-changes.md каждой области функций.

Файлы в области функций breaking-changes.md содержат список всех известных критических изменений для данного выпуска **,** а также историю критических изменений из прошлых выпусков.

Пример.

```md
Spatial sound breaking changes

2018.07.2
* Spatialization of the imaginary effect is now required.
* Management of randomized AudioClip files requires an entropy value in the manager node.

2018.07.1
No known breaking changes

2018.07.0
...
```

Сведения, содержащиеся в файлах breaking-changes.md уровня функций, будут объединены в заметки о выпуске для каждого нового выпуска МРТК.

Все критические изменения, которые являются частью изменения, **должны** быть документированы как часть запроса на включение внесенных изменений.

## <a name="tools-for-editing-markdown"></a>Средства для редактирования MarkDown

[Visual Studio Code](https://code.visualstudio.com/) — это отличное средство для редактирования файлов Markdown, которые являются частью документации по мртк.

При написании документации также настоятельно рекомендуется установить следующие два расширения:

- Документация расширения Markdown для Visual Studio Code — используйте сочетание клавиш ALT + M, чтобы открыть меню параметров создания документов.

- Средство проверки орфографии кода — слова с ошибками будут подчеркнуты. Щелкните правой кнопкой мыши слово с ошибками, чтобы изменить его или сохранить в словаре.

Оба эти пакета поставляются в пакет создания опубликованных документов Майкрософт.
