# <a name="how-to-use-sound-in-mixed-reality-applications"></a>Использование звука в приложениях смешанной реальности

Вы можете использовать звук для информирования и подкрепить пользовательскую модель состояния приложения. Используйте пространственность, если это уместно, чтобы поместить звуки в мир смешанной реальности. При таком способе подключения к подсистеме аудита и визуализации вы углубить интуитивно понятную природу взаимодействий и повысите уверенность пользователей.

## <a name="when-to-add-sounds"></a>Когда следует добавлять звуки

Приложения смешанной реальности часто имеют больше необходимости для звука, чем 2D-приложения, из-за отсутствия интерфейса тактиле. Добавляйте звуки, когда они сообщают пользователю или подкрепляются о взаимодействии.

### <a name="inform-and-reinforce"></a>Информирование и подкрепить

* Для событий, которые не инициируются пользователем, например уведомления, используйте звук для информирования пользователя о том, что произошло изменение.
* Взаимодействие может иметь несколько этапов. Используйте звук, чтобы усилить переходы по этапам.

См. следующие примеры взаимодействия, событий и характеристик звука.

### <a name="exercise-restraint"></a>Упражнение ограничения

У пользователей нет неограниченной емкости для звуковых данных.
* Каждый звук должен сообщать конкретные ценные сведения.
* Когда приложение воспроизводит звук для информирования пользователя, временно уменьшайте громкость других звуков.
* При наведении указателя на кнопку звук (см. следующую информацию) добавьте задержку для предотвращения чрезмерного срабатывания звука.

### <a name="dont-rely-solely-on-sounds"></a>Не полагайтесь только на звуки

Звуковые сигналы, которые хорошо используются, являются ценными для пользователей. Но убедитесь, что приложение можно использовать, даже если звук выключен.
* Пользователи могут столкнуться с нарушениями слуха.
* Приложение может использоваться в громкой среде.
* Пользователи могут столкнуться с вопросами конфиденциальности или другими причинами отключения звука устройства.

## <a name="how-to-sonify-interactions"></a>Как сонифи взаимодействия

К типам взаимодействия в смешанной реальности относятся жесты, прямые манипуляции и речь. Используйте следующие Рекомендуемые характеристики для выбора или проектирования звуков для этих взаимодействий.

### <a name="gesture-interactions"></a>Взаимодействия жестов

В смешанной реальности пользователи могут взаимодействовать с кнопками с помощью мыши. Действия кнопок обычно возникают, когда пользователь выдает, а не нажимает кнопку, чтобы дать пользователю возможность отменить взаимодействие. Используйте звуки, чтобы прикрепить эти этапы. Чтобы помочь пользователям в нацеливании на отдаленные кнопки, рассмотрите возможность использования звука с наведением указателя.
* Нажатие кнопки "звуки" должно быть коротким, тактиле "нажатием".<br/>Пример: [MRTK_ButtonPress. wav](https://github.com/microsoft/MixedRealityToolkit-Unity/tree/mrtk_development/Assets/MRTK/SDK/StandardAssets/Audio/MRTK_ButtonPress.wav)
* Кнопка — "нажимать" звуковое сопровождение должно иметь похожий вид тактиле. Более высокие тона, чем при нажатии клавиши "звук", применяют смысл завершения.<br/>Пример: [MRTK_ButtonUnpress. wav](https://github.com/microsoft/MixedRealityToolkit-Unity/tree/mrtk_development/Assets/MRTK/SDK/StandardAssets/Audio/MRTK_ButtonUnpress.wav)
* Для наведения звуков рассмотрите возможность использования тонкого и запугивание звука, например суд низкой частоты или выпуклости.

### <a name="direct-manipulation"></a>Непосредственное манипулирование

В HoloLens 2 отслеживание с обработкой с формулировкой поддерживает прямую манипуляцию с элементами пользовательского интерфейса. Звук имеет важное значение, если нет других физических отзывов.

Нажатие кнопки «звук *»* имеет большое значение, так как пользователь не получает никаких других индикаторов при достижении конца штриха. Звуковые индикаторы перемещения ключей могут быть небольшими, слабыми и перекрыто. Как и при взаимодействии с жестами, нажатия кнопки должны получить короткий, тактиленый звук, например щелчок. Нажатия клавиш должен иметь аналогичный звук щелчка, но с вызываемым шагом.
* Пример: [MRTK_ButtonPress. wav](https://github.com/microsoft/MixedRealityToolkit-Unity/tree/mrtk_development/Assets/MRTK/SDK/StandardAssets/Audio/MRTK_ButtonPress.wav)
* Пример: [MRTK_ButtonUnpress. wav](https://github.com/microsoft/MixedRealityToolkit-Unity/tree/mrtk_development/Assets/MRTK/SDK/StandardAssets/Audio/MRTK_ButtonUnpress.wav)

Трудно визуально подтвердить действие захвата или освобождения. Руки пользователя часто встречаются на любом визуальном элементе, а воплощающего объекты не имеют реального визуального аналога «захвата». Звук может эффективно обмениваться успешными взаимодействиями при захвате и выпуске.
* Действия при захвате должны иметь короткий, частично-муффлед тактиленый звук, который запрашивает идею пальцы, закрывающ вокруг объекта. Иногда есть также «ухая» звук, который ведет к извлечению звука, чтобы сообщить о движении руки.<br/>Пример: [MRTK_Move_Start. wav](https://github.com/microsoft/MixedRealityToolkit-Unity/tree/mrtk_development/Assets/MRTK/SDK/StandardAssets/Audio/MRTK_Move_Start.wav)
* Действия выпуска должны иметь похожий короткий и тактиленый звук. Обычно это более низкий тон, чем при захвате звука и в обратном порядке, а затем «ухая», чтобы сообщить, что объект связан с разделом.<br/>Пример: [MRTK_Move_End. wav](https://github.com/microsoft/MixedRealityToolkit-Unity/tree/mrtk_development/Assets/MRTK/SDK/StandardAssets/Audio/MRTK_Move_End.wav)

Взаимодействие с *рисованием* должно получить постоянный и циклический перебор звука с объемом, определяемым перемещением руки пользователя. Он должен быть скрытым, когда рука пользователя остается и громкее, когда рука перемещается быстро.

### <a name="voice-interactions"></a>Голосовое взаимодействие

Речевое взаимодействие часто имеет небольшие визуальные элементы. Используйте звуки, чтобы усилить этапы взаимодействия. Вы можете использовать дополнительные тональные звуки, чтобы отличать их от жестов и звуков с прямым управлением.

* Используйте положительный звуковой сигнал для *подтверждения* команд голоса. Эффективны и основные музыкальные интервалы.
* Используйте более короткий и менее положительный звуковой сигнал при *сбоях* команд речи. Избегайте отрицательных звуков. Вместо этого используйте более перкуссиве независимый звук, чтобы сообщить, что приложение перемещается из взаимодействия.
* Если в приложении есть пробуждение, используйте короткий, добройный тон, когда устройство *начинает прослушивание*. Используйте слабый циклический звук во *время прослушивания* приложения.

### <a name="notifications"></a>Уведомления

Уведомления сообщают об изменениях состояния приложения и других событиях, которые пользователь не инициировал. Изменения состояния могут включать завершения процессов, сообщения и телефонные звонки.

В смешанной реальности объекты иногда переходят из поля зрения пользователя. Связывание перемещения *анимированных объектов* с пространственными звуками, которые зависят от типа объекта и скорости движения.
* Он позволяет воспроизводить Пространственный звук в конце анимации, чтобы информировать пользователя о новой позиции объекта.
* Для постепенных движений в процессе перемещения ухая звук, который помогает пользователю отслеживанию объекта.

Звуки *уведомлений о сообщениях* могут слышаться многократно, иногда в быстром последующем. Важно, чтобы они не выхаршаись и не выделялись звуковым сигналом. Положительные тональные звуки среднего радиуса вступают в силу.

* Звуки входящих вызовов должны иметь аналогичные качества для телефонной мелодии звонка. Эти звуки представляют собой циклические фразы, которые воспроизводятся до тех пор, пока пользователь не ответит на звонок.
* В связи с голосовым подключением и отсоединении должен быть короткий тональный звук. Звук подключения должен быть положительным тоном, чтобы указать на успешное подключение. Звук для отключения должен быть нейтральным звуком, указывающим на завершение вызова.

## <a name="handle-spatialization"></a>Работа с пространственными данными

Для размещения звуковых сигналов в мире смешанной реальности при пространственной связи используются Стереофонические наушники или динамики.

### <a name="which-sounds-to-spatialize"></a>Какие звуки спатиализе

Звук должен быть пространственно, если он связан с событием с пространственным расположением. Сюда входят пользовательский интерфейс, включенные голоса AI и визуальные индикаторы.

Спатиализе элементы *пользовательского интерфейса* , чтобы избавиться от закачки пользователя, ограничивая число стереозвуков, которые они прослушивают. Такие взаимодействия, как касание, выделение и освобождение, кажутся более естественными при пространственной обратной связи. Примите во внимание следующие сведения о ослаблении расстояния для этих элементов.

Спатиализе *визуальные индикаторы* и ПРИносят *искусственные голоса* , чтобы интуитивно информировать пользователей, когда эти вещи выходят за рамки представления.
    
В отличие от этого, старайтесь не использовать пространственность для *AI-голоса без распознавания лиц* и других элементов, у которых нет четко определенного пространственного расположения. Пространственное отсутствие связанного визуального элемента может помешать пользователям подумать о том, что не удается найти визуальный элемент.

При использовании пространственной связи с некоторыми затратами на ресурсы ЦП. Во многих приложениях одновременно воспроизводятся не более двух звуков. В этом случае стоимость пространственности может быть незначительной. С помощью монитора частоты кадров МРТК можно оценить влияние добавления пространственности.

### <a name="when-and-how-to-apply-distance-based-attenuation"></a>Когда и как применять ослабление на основе расстояния

В физическом мире звуки, которые находятся дальше, находятся в тихом направлении. Ваш модуль аудио может моделировать это ослабление на основе исходного расстояния. Используйте ослабление на основе расстояния при обмене релевантной информацией.

Расстояние до *визуальных индикаторов*, *анимированных голограмм* и других информативных звуков относится к пользователю. Используйте ослабление на основе расстояния, чтобы интуитивно предоставить подсказки.

Настройте кривую затухания для каждого источника в соответствии с размером сфер мира смешанной реальности. Кривая по умолчанию для звукового модуля обычно предназначена для больших (до половины километра) пробелов.

Звук, получающий *прогрессивные этапы действий кнопок* и другие взаимодействия, не должен применять ослабление. Эффект от этих звуков является более важным, чем связь расстояния с кнопкой. Вариации могут отвлечь друг друга, особенно с клавиатуры, когда много щелчков кнопок могут быть продолжены.

### <a name="which-spatialization-technology-to-use"></a>Какую технологию пространственного использования

С наушниками или с помощью головных динамиков HoloLens используйте технологии пространственной связи на основе функций перенаправления (ХРТФ). Эти технологии моделируют распространение звука по всему миру в физическом мире. Даже если источник звука находится на дальней стороне одного из этих головок, звук будет распространен на отдаленное, а с некоторой затуханием и задержкой. Панорамирование докладчика полагается только на ослабление и применяет общую затухание в левом конце, когда звук располагается справа, а другой способ — вокруг. Эта методика может быть неудобной для прослушивателей "нормальных слухи" и недоступна для прослушивателей с нарушениями слуха за один год.
